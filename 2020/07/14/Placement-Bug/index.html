Placement Bug

Posted on 2020-07-14

Pike 版 Placement bug 案例

问题描述：

因要停机维护物理机，为不影响虚机业务，将虚机热迁移到其他节点报错：“No valid host was found ”。
因为我是定点迁移，所以目标节点的资源是足够的，排除资源不够导致报错的可能性，查看日志有如下报错：

截图中红色标注的意思是多，有个虚机已经移动到compute10节点了，但是之前的所在的原节点的资源并没有释放。因此要解决这个问题，需要想办法释放被占用的资源。

这里先贴解决过程：

1.查看问题主机在各个节点中的资源分配情况


可以看到同一台主机，在两个节点上有资源分配，正常情况下应该只有一个资源分配节点。主机id为3b开头的是正确的物理节点。

2.确认好目前问题主机所在的节点，然后将未释放资源的节点进行清除。

运行命令：openstack resource provider allocation set 主机id –allocation rp=节点id（正确的）,VCPU=20,MEMORY_MB=65536，执行完命令后，再次查看资源分配情况：之前被占用的节点分配已经清除。

上面用到的命令中，诸如resource、provider等都是placement组件里的概念，那么什么是Placement？

Placement

官方的定义：让用户和代码能够便捷的使用、管理、监控整个 OpenStack 的系统资源，这就是 Placement（布局）。
Placement最早在Newton版本中被引入，而在Pike版本中，Placement API服务是被要求强制启动的，否则的话你的nova-compute service是无法正常运行的。在Stein版本中，Placement成为独立项目，并发布第一个正式版本1.0.0
